# ADR-002: v0.3.0 — Behavioral Detection + MCP Security Analyzer

- **Status:** DRAFT
- **Date:** 2026-04-05
- **Author:** thc1006
- **Grounded in:** Web research conducted 2026-04-05 (3 parallel research tracks)


## Context

### What v0.2.0 got wrong

ShiftScope v0.2.0 shipped with 5 analyzers and 26 rules, but real-world validation against 22 public Ingress manifests revealed that **85% of findings were `gw-annotation-unknown`** — the tool was correctly identifying annotations but providing no actionable guidance.

The root cause: we focused on annotation mapping (what ingress2gateway v1.0 already does for 30+ annotations) instead of what ingress2gateway does NOT do.

### What the real world actually needs (April 2026)

**Source:** Kubernetes official blog (2026-02-27), Hacker News practitioner threads, DEV Community migration war stories, Astrix/Endor Labs security surveys.

#### 1. Gateway API behavioral differences that silently break traffic

The Kubernetes blog "Before You Migrate" (2026-02-27) documents 5 behavioral differences that cause silent 404s and routing failures:

| Behavior | Impact | ingress2gateway handles? |
|----------|--------|------------------------|
| Regex is prefix-based + case-insensitive in nginx, full + case-sensitive in GW API | **Silent 404s** | No |
| `use-regex` applies globally per host across ALL Ingresses | **Silent 404s on unrelated Ingresses** | No |
| `rewrite-target` silently enables regex mode for entire host | **Silent 404s** | No |
| Missing trailing slash gets auto-301 in nginx, 404 in GW API | **Broken client redirects** | No |
| URL path normalization (`..`, `//`) differs per implementation | **Inconsistent routing** | No |

Plus 5 additional community-reported issues: ReferenceGrant silent failures, snippet no-equivalent, canary translation gaps, session affinity differences, host merge conflicts.

#### 2. MCP security crisis

Endor Labs survey (2,614 MCP implementations): **82% vulnerable to path traversal, 67% to code injection, 34% to command injection.** 30+ CVEs in 60 days. Astrix survey (5,200+ implementations): **53% use static API keys, only 8.5% use OAuth.**

CVE-2026-32211 (Azure MCP Server): CVSS 9.1, zero authentication, exposes API keys and project data.

**No existing ShiftScope analyzer addresses MCP security.** The agent-readiness analyzer checks tool allowlists and token budgets but NOT the security posture of MCP server implementations.

#### 3. Agent token cost blindspot

Average agent monthly operating cost: $3,200–$13,000. One organization discovered a $14,000/month agent only through token accounting. 80% of organizations deploying autonomous AI cannot tell what those systems are doing in real time.

OpenTelemetry GenAI semantic conventions (experimental, March 2026) define standard spans for LLM calls, tool executions, and agent decision flows — but most deployments don't implement them.

## Decision

### v0.3.0 scope: two new analyzer upgrades + one new analyzer

#### A. Gateway API Analyzer v2: Behavioral Detection Rules

Upgrade the existing gateway-api analyzer from "annotation mapping" to "behavioral risk detection":

| Rule ID | Detection | Severity | Source |
|---------|-----------|----------|--------|
| `gw-behavior-regex-prefix` | `use-regex: "true"` + regex missing `.*` or `(?i)` | CRITICAL | K8s blog behavior #1 |
| `gw-behavior-regex-global` | Multiple Ingresses sharing host, any has `use-regex` | CRITICAL | K8s blog behavior #2 |
| `gw-behavior-rewrite-implies-regex` | `rewrite-target` present + sibling Ingresses on same host | CRITICAL | K8s blog behavior #3 |
| `gw-behavior-trailing-slash` | Non-regex path ending in `/` | HIGH | K8s blog behavior #4 |
| `gw-behavior-path-normalization` | Universal warning on all Ingresses | MEDIUM | K8s blog behavior #5 |
| `gw-behavior-cross-namespace` | Backend references different namespace | HIGH | Community |
| `gw-behavior-snippet-no-equiv` | `configuration-snippet` or `server-snippet` | CRITICAL | Community |
| `gw-behavior-canary` | `canary-*` annotations | HIGH | Community |
| `gw-behavior-affinity` | `affinity` annotation | MEDIUM | Community |
| `gw-behavior-host-merge` | Multiple Ingresses on same host | MEDIUM | Community |

**Key design decision:** Rules 1-3 require **cross-Ingress analysis** — grouping all Ingresses by hostname and checking interactions. This is a fundamentally different pattern from v0.2.0's per-Ingress analysis.

**Implementation:** The parser must return all Ingresses from a manifest, and the analyzer must group them by hostname before running behavioral rules.

#### B. New: MCP Security Analyzer

A completely new analyzer targeting the 12 known MCP vulnerability patterns:

| Rule ID | Detection | Severity | OWASP ASI |
|---------|-----------|----------|-----------|
| `mcp-sec-tool-poisoning` | Scan tool descriptions for injection keywords, Base64, Unicode smuggling | CRITICAL | ASI-01 |
| `mcp-sec-path-traversal` | File tools without path canonicalization | CRITICAL | ASI-02 |
| `mcp-sec-command-injection` | `shell=True`, `exec()`, `eval()`, string-concatenated commands | CRITICAL | ASI-05 |
| `mcp-sec-code-injection` | Template literal injection, unsanitized code generation | CRITICAL | ASI-05 |
| `mcp-sec-missing-auth` | No OAuth/mTLS config, binding to 0.0.0.0 | CRITICAL | ASI-03 |
| `mcp-sec-static-credentials` | Plaintext API keys in config, env vars, hardcoded tokens | HIGH | ASI-03 |
| `mcp-sec-prompt-injection` | Control characters, instruction patterns in tool inputs | CRITICAL | ASI-01 |
| `mcp-sec-ssrf` | HTTP clients without URL allowlisting | HIGH | ASI-02 |
| `mcp-sec-over-permission` | Wildcard permissions, excessive tool count per agent | HIGH | ASI-03 |
| `mcp-sec-supply-chain` | Package name typosquatting, missing version pins | HIGH | ASI-04 |

**Input format:** MCP server config files (`mcp.json`, `claude_desktop_config.json`, `.cursor/mcp.json`) + optional source code scanning.

**Competitive landscape:** MCPScan.ai, Snyk Agent Scan, Invariant MCP-Scan, Enkrypt AI exist but are either commercial, LLM-dependent, or narrow. ShiftScope's MCP security analyzer is **deterministic-first** (rule-based, no LLM required) and **pluggable** (new rules via SDK).

#### C. Agent Readiness Analyzer v2: Cost + OTel Governance

**Grounding:** $47,000 LangChain infinite loop incident (Nov 2025) — two agents ping-ponged for 11 days with no cost tracking, no loop detection, no budget limits. 96% of enterprises report AI costs exceeding initial estimates. OTel GenAI semantic conventions (experimental, March 2026) define standard spans but NO cost metrics — cost calculation must be derived from token counts × pricing tables.

**Three-layer governance model** (from Cordum's Agent FinOps Guide + agentgateway docs):
- Layer 1: Per-action (tokens/call, max runtime seconds)
- Layer 2: Per-agent (budget/time window, pause on exhaust)
- Layer 3: Fleet-level (throttle at 80%, pause all at 95%)

Extended rules:

| Rule ID | Detection | Severity | Source |
|---------|-----------|----------|--------|
| `agent-cost-no-budget` | No `max_cost_usd`, `max_tokens`, or `max_runtime_sec` | CRITICAL | $47K incident |
| `agent-cost-no-loop-guard` | No `max_iterations` or loop detection configured | CRITICAL | $47K incident |
| `agent-cost-unbounded-retry` | No exponential backoff / kill-after-N limit | HIGH | 7 failure modes |
| `agent-cost-expensive-routing` | Low-complexity tasks routed to expensive models | WARNING | 30-50x cost diff |
| `agent-otel-missing-required` | Missing `gen_ai.client.operation.duration` (required by spec) | HIGH | OTel spec |
| `agent-otel-no-token-tracking` | Missing `gen_ai.usage.input_tokens` / `output_tokens` on spans | HIGH | OTel spec |
| `agent-otel-no-agent-id` | Missing `gen_ai.agent.id` / `gen_ai.agent.name` | WARNING | OTel spec |
| `agent-governance-no-kill-switch` | No termination mechanism configured | CRITICAL | OWASP ASI-10 |
| `agent-governance-no-audit-trail` | No cost evidence records emitted per action | HIGH | FinOps best practice |
| `agent-governance-no-graduated-response` | No 75%/90%/100% budget threshold actions | WARNING | agentgateway pattern |

### Technology choices for v0.3.0

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Cross-Ingress analysis | Group-by-hostname in parser, pass all Ingresses to rules | Required for behaviors #1-3 |
| MCP config parsing | JSON parser for `mcp.json` family | Standard format |
| MCP source scanning | Optional, regex-based static analysis | Keep deterministic-first |
| OWASP ASI mapping | Map each MCP finding to ASI-01 through ASI-10 | Compliance alignment |
| Agent cost metrics | Parse OTel config + agent config JSON | No runtime dependency |

**v0.3.0 total new rules: 30** (10 behavioral + 10 MCP security + 10 agent governance v2)

### What we explicitly DO NOT do

1. **We do not replace ingress2gateway** — they do format conversion, we do risk detection
2. **We do not require an LLM** for MCP scanning — deterministic rules, not AI-dependent
3. **We do not scan running MCP servers** — static config/source analysis only
4. **We do not compete with Microsoft Agent Governance Toolkit** — they do runtime enforcement, we do pre-deployment analysis

## Consequences

### Positive
1. Gateway API analyzer becomes genuinely useful for the 50% of clusters in migration crisis
2. MCP Security analyzer fills the biggest unmet need in the agent governance space
3. OWASP ASI mapping provides compliance value
4. All three upgrades are deterministic-first, consistent with ShiftScope's core philosophy

### Negative
1. Cross-Ingress analysis adds complexity to the parser/analyzer contract
2. MCP security is a fast-moving space — rules need frequent updates
3. Source code scanning (even regex-based) has false positive risk

### Risks
| Risk | Mitigation |
|------|-----------|
| Cross-Ingress grouping changes analyzer API | Backward-compatible: single-Ingress manifests still work |
| MCP config format varies across clients | Start with top 3: Claude Desktop, Cursor, VS Code |
| False positives in source scanning | Default to config-only scanning; source scanning opt-in |
